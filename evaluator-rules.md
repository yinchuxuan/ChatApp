| 维度 | 问题 | 分数 (0-10) | 备注 |
| --- | --- | --- | --- |
| 正确性 | 实现出来的行为是否符合目标功能？ |  |  |
| 验证 | 要求的检查是否真的跑过，并留下证据？ |  |  |
| 设计 | 当前实现的设计(代码架构，前端设计)是否最佳，品味如何？ ｜ ｜ ｜

| 可维护性 | 代码和文档是否清楚到足以交给下一轮会话？ |  |  |

## 评估原则

- 阅读feature_list.json中对应的工作项，严格审查verification中各项要求是否完成
- 用最严格的标准评估worker的工作，禁止轻易给其工作打满分
- 对于设计要有高标准的要求，对于混乱、不优雅、不合理的代码实现以及不美观的ui实现，即使功能正确实现也要拒绝
- 禁止worker进行git代码提交
- 仓库中所有文件不应该超过200行
