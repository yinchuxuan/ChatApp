# Evaluation Scorecard

Complete a review using this scorecard after implementation and before formal acceptance.

| Dimension | Question | Score (0-10) | Notes |
| --- | --- | --- | --- |
| Correctness | Does the implemented behavior match the target feature? |  |  |
| Verification | Were the required checks actually run, with evidence left behind? |  |  |
| Design | Is the current implementation's design optimal? How is the taste? |  |  |
| Scope Discipline | Did this round mostly stay within the scope of the selected feature? |  |  |
| Maintainability | Is the code and documentation clear enough to be handed off to the next session? |  |  |

## Evaluation Principles

- Read the corresponding work item in `auto_mode_harness/feature_list.json` and strictly review whether all verification requirements are met
- Evaluate the worker's work with the strictest standards; never give a perfect score easily
- Hold design to a high standard: reject work with messy, inelegant, or unreasonable code
- The worker is prohibited from performing git commits
- All files in the repository must not exceed 200 lines
